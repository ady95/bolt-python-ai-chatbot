# 파일 첨부 메시지 처리 수정사항

## 1. 배경 및 문제

기존 [listeners/events/app_messaged.py](listeners/events/app_messaged.py)는 Slack 메시지 이벤트를 처리할 때 아래와 같이 `subtype`이 존재하기만 하면 무조건 이벤트를 무시하고 있었다.

```python
# (기존 코드)
if event.get("bot_id") or event.get("subtype") is not None:
    return
```

주석상 의도는 "편집/입퇴장/삭제 같은 noise성 이벤트와 bot loop 차단"이었지만, Slack은 사용자가 파일을 업로드할 때 **`subtype: "file_share"`** 로 메시지 이벤트를 보낸다. 따라서 파일이 첨부된 메시지는 모두 이 필터에 걸려 AI 응답 로직까지 도달하지 못했고, 봇이 아무런 반응도 하지 않는 것처럼 보였다.

## 2. 요구사항

1. 파일이 첨부된 메시지에 대해서는 AI 처리는 하지 말고, 첨부된 파일만 다운로드해서 프로젝트 루트의 `download/` 폴더에 저장할 것.
2. 다운로드 결과에 따라 원본 메시지에 이모지 리액션을 붙일 것.
   - 성공: `:white_check_mark:` (흰색 확인 표시)
   - 실패: `:x:`

## 3. 변경 내용 요약

수정 파일: [listeners/events/app_messaged.py](listeners/events/app_messaged.py)

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| subtype 필터 | 모든 subtype 차단 | `file_share`만 통과, 나머지는 계속 차단 |
| 파일 첨부 처리 | (없음 — 이벤트 자체가 차단됨) | `download/`에 저장 후 조기 return |
| 파일 메타데이터 조회 | — | 이벤트 payload가 스텁이면 `files.info` API로 재조회 |
| 결과 피드백 | — | 원본 메시지에 ✅ / ❌ 리액션 추가 |
| AI provider 호출 | 모든 통과 메시지에 대해 수행 | 파일 첨부 메시지에는 **수행하지 않음** |
| 추가 의존성 | — | 표준 라이브러리만 사용 (`os`, `urllib.request`) |

## 4. 상세 변경 내역

### 4.1 import 및 상수 추가

```python
import os
import urllib.request

# ...

# 프로젝트 루트의 download/ 폴더를 절대경로로 계산.
# 이 파일은 listeners/events/ 아래에 있으므로 두 단계 상위가 레포 루트.
_DOWNLOAD_DIR = os.path.normpath(
    os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", "..", "download")
)
```

- `__file__` 기준으로 경로를 계산하므로 프로세스의 현재 작업 디렉터리(CWD)에 의존하지 않는다.
- `os.path.normpath`로 경로를 정규화한다.

### 4.2 다운로드 헬퍼 함수 (스텁 감지 포함)

```python
def _download_slack_file(file_info: dict, client: WebClient, token: str, dest_dir: str):
    """Download a single Slack file object to ``dest_dir``.

    Returns the saved absolute path, or ``None`` if the file had no download URL.
    Slack's private file URLs require a bearer token on the request.

    The ``files`` entries in a ``message`` event are often only stubs (see
    ``file_access == 'check_file_info'``) without ``url_private``. In that case we
    fetch the full file metadata via ``files.info`` before downloading.
    """
    url = file_info.get("url_private_download") or file_info.get("url_private")

    # 스텁 감지: Slack은 이벤트 payload에 파일 메타데이터를 축약해서 보낼 때가 있고,
    # 이 경우 files.info를 별도로 호출해서 실제 URL을 얻어와야 한다.
    if not url or file_info.get("file_access") == "check_file_info":
        file_id = file_info.get("id")
        if not file_id:
            return None
        info_resp = client.files_info(file=file_id)
        full_info = info_resp.get("file") or {}
        # 새로 가져온 값으로 스텁을 덮어쓴다.
        file_info = {**file_info, **full_info}
        url = file_info.get("url_private_download") or file_info.get("url_private")
        if not url:
            return None

    file_id = file_info.get("id") or ""
    name = file_info.get("name") or f"{file_id or 'file'}.bin"
    # 동시 업로드 파일명 충돌을 피하기 위해 file_id를 접두사로 사용.
    safe_name = f"{file_id}_{name}" if file_id else name
    dest_path = os.path.join(dest_dir, safe_name)
    req = urllib.request.Request(url, headers={"Authorization": f"Bearer {token}"})
    with urllib.request.urlopen(req) as resp, open(dest_path, "wb") as out:
        out.write(resp.read())
    return dest_path
```

핵심 포인트:

- **스텁(stub) 처리**: 실제 운영 중 확인된 이슈로, Slack의 `message` 이벤트 payload에 실리는 `files` 배열은 종종 아래처럼 최소한의 메타데이터만 담긴 "스텁" 형태로 온다.
  ```json
  {"id": "F0ASBJ32DE0", "file_access": "check_file_info",
   "created": 0, "timestamp": 0, "user": "", "filetype": "pdf"}
  ```
  이 경우 `url_private` / `url_private_download` 필드가 **아예 없어서** 바로 다운로드할 수 없다. `file_access == "check_file_info"` 플래그가 바로 "files.info를 호출해서 진짜 메타데이터를 가져가라"는 신호다. 헬퍼는 이 플래그나 URL 부재를 감지하면 `client.files_info(file=file_id)`로 전체 메타데이터를 재조회한 뒤 다운로드를 진행한다.
- **인증 헤더**: Slack의 private 파일 URL은 인증이 필요하므로 **`Authorization: Bearer <bot_token>`** 헤더를 반드시 붙여야 한다.
- **URL 우선순위**: `url_private_download`가 있으면 우선 사용하고, 없으면 `url_private`로 폴백.
- **파일명 규칙**: 저장 파일명은 `{file_id}_{원래파일명}` 형식으로 동시 업로드 시의 충돌을 방지한다.
- **스킵 반환**: 다운로드 URL을 끝내 얻지 못한 파일(일부 특수 타입 등)은 `None`을 반환하여 호출 측에서 실패로 집계한다.

### 4.3 subtype 필터 완화

```python
# 봇 메시지는 무한 루프 방지를 위해 차단.
if event.get("bot_id"):
    return

# 편집/삭제/입퇴장 등 noise성 subtype은 계속 차단하되,
# file_share만 통과시켜 첨부파일을 저장할 수 있게 한다.
subtype = event.get("subtype")
if subtype is not None and subtype != "file_share":
    return
```

- 봇 메시지 체크와 subtype 체크를 분리해 가독성을 높였다.
- 화이트리스트가 아닌 "예외 허용" 방식을 택해, 새로운 noise subtype이 생기더라도 자동으로 차단되도록 했다.

### 4.4 파일 첨부 분기 처리 (리액션 피드백 포함)

```python
# 메시지에 파일이 첨부되어 있으면 download/ 폴더에 저장하고
# AI 처리는 건너뛴다. 함께 온 텍스트도 이 경로에서는 무시된다.
files = event.get("files") or []
if files:
    message_ts = event.get("ts")
    all_succeeded = True
    try:
        os.makedirs(_DOWNLOAD_DIR, exist_ok=True)
        token = client.token
        for f in files:
            try:
                saved_path = _download_slack_file(f, client, token, _DOWNLOAD_DIR)
                if saved_path:
                    logger.info(f"Saved Slack file to {saved_path}")
                else:
                    all_succeeded = False
                    logger.warning(
                        f"Skipped Slack file {f.get('id')}: no download URL"
                    )
            except Exception as file_err:
                all_succeeded = False
                logger.error(
                    f"Failed to download Slack file {f.get('id')}: {file_err}"
                )
    except Exception as e:
        all_succeeded = False
        logger.error(f"Error while handling file upload: {e}")

    # 성공/실패 결과에 따라 원본 메시지에 이모지 리액션을 붙인다.
    if channel_id and message_ts:
        reaction_name = "white_check_mark" if all_succeeded else "x"
        try:
            client.reactions_add(
                channel=channel_id, name=reaction_name, timestamp=message_ts
            )
        except Exception as react_err:
            logger.error(f"Failed to add reaction {reaction_name}: {react_err}")
    return
```

주요 설계 결정:

- **개별 파일 오류 격리**: 한 파일의 다운로드가 실패해도 루프가 중단되지 않고 나머지 파일을 계속 처리한다.
- **성공/실패 플래그 (`all_succeeded`)**: 루프 진입 전에 `True`로 시작하고, 다운로드 실패/스킵/초기화 오류가 하나라도 발생하면 `False`로 내린다. 즉 **부분 실패도 실패로 간주**한다.
- **리액션 피드백**: 루프 종료 후 원본 메시지(`event["ts"]`)에 `client.reactions_add`를 호출해 이모지를 붙인다.
  - 전부 성공 → `:white_check_mark:`
  - 하나라도 실패 → `:x:`
- **리액션 호출 실패 격리**: 리액션 API 호출 자체가 실패해도 로깅만 하고 예외를 전파하지 않는다 (예: `reactions:write` scope 부재).
- **AI 응답 없음**: 사용자 요구사항대로 로딩 메시지 / AI 응답 메시지는 일절 보내지 않는다. 결과는 오직 리액션 + 서버 로그로 전달된다.
- **조기 return**: 파일이 있으면 이후 AI provider 호출, `chat_update` 등을 전부 건너뛴다.

## 6. 저장 파일 경로

```
{프로젝트 루트}/download/{file_id}_{원본파일명}
```

예시:
```
d:\DEV_slack\bolt-python-ai-chatbot\download\F0123ABCD_report.pdf
d:\DEV_slack\bolt-python-ai-chatbot\download\F0456EFGH_screenshot.png
```

`download/` 폴더는 실행 시점에 `os.makedirs(..., exist_ok=True)`로 자동 생성된다.

## 7. 필수 권한 (Slack Scopes)

Slack 앱의 **Bot Token Scopes** 에 다음이 반드시 설정되어 있어야 한다.

| Scope | 용도 | 없을 때 증상 |
|---|---|---|
| `files:read` | private 파일 메타데이터 조회(`files.info`) 및 다운로드 URL 접근 | `missing_scope` 에러, 다운로드 시 HTTP 401/403 |
| `reactions:write` | 원본 메시지에 ✅/❌ 리액션 추가 | `missing_scope` 에러 로그 (파일 저장은 정상 동작) |
| `channels:history` / `groups:history` / `im:history` / `mpim:history` | `message` 이벤트 수신 | 이벤트 자체가 도착하지 않음 |

⚠️ **중요**: scope을 추가하거나 변경한 뒤에는 반드시 Slack 앱 설정 페이지 상단의 **Reinstall to Workspace** 버튼을 눌러 앱을 재설치해야 한다. 재설치하지 않으면 새 scope이 토큰에 반영되지 않는다.

## 8. 트러블슈팅: 실제 겪은 이슈 기록

### 8.1 수정 코드가 반영되지 않음

**증상**: `app_messaged.py`를 수정했는데도 앱 재실행 후 동작이 동일함.

**원인**: [listeners/events/__init__.py](listeners/events/__init__.py)가 백업본인 `app_messaged_ori`에서 콜백을 import하고 있었다.

```python
# 잘못된 상태
from .app_messaged_ori import app_messaged_callback
```

**해결**: 현재 작업 파일인 `app_messaged`에서 import하도록 수정.

```python
from .app_messaged import app_messaged_callback
```

### 8.2 파일 이벤트는 들어오는데 다운로드가 스킵됨

**증상**: 콘솔에 `Received message event: ...` 로그는 찍히지만 `download/` 폴더에 아무것도 저장되지 않음.

**원인**: 이벤트 payload의 `files` 배열이 `file_access: "check_file_info"` 플래그가 붙은 스텁이어서 `url_private` 필드가 없었다. 초기 헬퍼는 URL이 없으면 그대로 `None`을 반환해 스킵하고 있었다.

**해결**: 4.2 섹션의 스텁 감지 로직을 추가해, 스텁이 감지되면 `client.files_info(file=file_id)`로 메타데이터를 재조회하도록 변경.

## 9. 한계 및 향후 개선점

1. **대용량 파일**: 현재 구현은 `resp.read()`로 응답 전체를 메모리에 올린 뒤 파일로 쓴다. 수백 MB 이상의 파일이 예상된다면 chunk 스트리밍 방식(`shutil.copyfileobj`)으로 변경 필요.
2. **동기 다운로드**: `urllib.request.urlopen`은 블로킹 호출이며, 여러 파일이 첨부된 경우 순차적으로 처리된다. 처리량이 문제되면 `ThreadPoolExecutor` 병렬화 고려.
3. **부분 실패 구분 불가**: 여러 파일 중 일부만 실패해도 메시지에는 `:x:` 한 번만 붙는다. 어느 파일이 실패했는지는 현재 서버 로그로만 확인 가능하다. 필요하다면 스레드 댓글로 실패한 파일명을 남기는 방식 고려.
4. **중복 업로드 처리**: 동일 `file_id`는 항상 같은 파일명으로 저장되므로, 같은 파일이 두 번 전달되면 기존 파일을 덮어쓴다. 이력 보존이 필요하다면 타임스탬프를 파일명에 포함시킬 것.
5. **파일 타입 필터 없음**: 모든 타입(문서/이미지/비디오/스니펫 등)을 동일하게 저장한다. 특정 타입만 저장하려면 `file_info.get("filetype")` 또는 `mimetype`으로 필터링 가능.
6. **files.info 추가 호출 비용**: 스텁 감지 시마다 API 호출이 한 번 더 일어난다. 업로드 빈도가 매우 높다면 rate limit에 유의.

## 10. 테스트 체크리스트

- [ ] DM에서 일반 텍스트 입력 → AI 응답 정상, 리액션 없음
- [ ] DM에서 파일 업로드 → `download/`에 저장, AI 응답 없음, ✅ 리액션 부여
- [ ] DM에서 파일 + 텍스트 동시 업로드 → `download/`에 저장, 텍스트는 AI로 보내지 않음, ✅ 리액션
- [ ] 채널에서 파일 업로드 → `download/`에 저장, ✅ 리액션
- [ ] 채널에서 `@봇` 멘션만 → `app_mention` 핸들러로 처리 (중복 응답 없음)
- [ ] 메시지 편집 → 아무 동작 없음
- [ ] 여러 파일 동시 업로드 (전부 정상) → 모든 파일 저장, ✅ 리액션 1회
- [ ] 여러 파일 중 일부 실패 상황 → 성공분만 저장, ❌ 리액션
- [ ] `files:read` scope 제거 후 테스트 → `download/`에 저장 안 됨, ❌ 리액션, 서버 로그에 `missing_scope`
- [ ] `reactions:write` scope 제거 후 테스트 → 파일은 정상 저장, 리액션만 실패하며 서버 로그에 에러
- [ ] 저장 경로 확인: `d:\DEV_slack\bolt-python-ai-chatbot\download\`
- [ ] 서버 로그에 `Saved Slack file to ...` 출력 확인
